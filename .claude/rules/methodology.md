# coding-vibe 核心方法论

## 核心理念

与 AI 协作开发时，**副驾模式优于自动驾驶模式**。人类决定方向，AI 负责落地、验证和查漏。

## 三层决策模型

| 层 | 角色 | 负责 |
|----|------|------|
| **战略** | 人类 | 需求定义、架构决策、验收标准 |
| **战术** | AI | 实现方案、代码生成、测试覆盖 |
| **验证** | AI+人类 | 审查、测试、部署验证 |

## 工作流循环

```
需求 → 规划 → 实现 → 验证 → 反馈 →（循环）
```

### 1. 需求分析
- 人类描述目标（功能/修复/优化）
- AI 复述理解，确认范围
- 拆解为可执行子任务

### 2. 规划
- AI 输出实施方案（涉及哪些文件、改动点、风险）
- 大型改动 → Plan 模式 / 子代理

### 3. 实现
- 遵循项目的场景模式优先序
- 小步改动，频繁验证
- 优先使用已有工具和模式

### 4. 验证
- 自动化测试（单元/集成/E2E）
- 代码审查（AI 自审 + 人类审）
- 手动验证关键路径

### 5. 反馈
- 测试失败 → 修复并重新验证
- 方向偏差 → 及时纠正
- 合并代码 → 原子 commit

## 信息分级策略

| 等级 | 适用范围 | 行动 |
|------|----------|------|
| 已验证 | 本轮工具确认过的信息 | 直接使用，autocompact 后重新验证 |
| 高置信 | 语言规范、稳定 API、标准库 | 可使用，被质疑时查证 |
| 需查证 | 新库、新 API、记忆中的内容 | 先查证再使用 |

## 代码原则

1. **不可变性（Immutability）**：永远创建新对象，不修改现有对象
2. **小文件原则**：单个文件 ≤ 800 行，200-400 行为佳
3. **边界验证**：所有输入在系统边界验证（用户输入、API 响应、文件内容）
4. **显式错误处理**：不静默吞掉错误，UI 层给友好提示，服务端记详细日志
5. **功能组织**：按功能/领域组织，不按类型堆叠

## 输出纪律

1. 所有验证过的信息才能输出
2. 提及文件/函数前先 Read 确认内容
3. 不确定时明确说不确定，不编造接口参数
4. 发现之前有误 → 立即承认并更正

---

*这套方法论是活的，随着实践持续演进。如果发现更好的模式，更新规则后通过 PR 审查合并。*
